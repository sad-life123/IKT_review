# IKT Review Architecture

## Назначение

Плагин `local_ikt_review` собирает показатели использования ИКТ по курсам
дневной формы обучения в Moodle. Один запуск формирует отчетный снимок за
выбранный период, рассчитывает метрики по каждому курсу и сохраняет результат
для последующего сравнения.

Проект ориентирован на PostgreSQL. Collect SQL использует возможности
PostgreSQL: `FILTER`, `ON CONFLICT`, partial indexes и covering indexes через
`INCLUDE`.

Источник бизнес-формул: `Показатели эффективности использования ИКТ-2.pdf`.

## Пользовательские страницы

У плагина одна основная страница и один JSON endpoint подробностей запуска.

### Основная страница

`/local/ikt_review/index.php`

- ставит полный production-расчет в очередь Moodle ad hoc tasks;
- показывает сводку последнего завершенного production-запуска;
- показывает историю production-запусков;
- открывает модальное окно запуска с диаграммами и разрезом по кафедрам;
- не показывает старые `selected` и `synthetic` прогоны, если они остались в
  базе после удаленной временной диагностической зоны.

### Подробности запуска

`/local/ikt_review/details.php`

- возвращает HTML для модального окна live-страницы;
- показывает контрольные показатели, гистограммы элементов, распределение
  модулей и разрез по кафедрам;
- доступен только для завершенных запусков.

## Модель данных

Плагин хранит четыре таблицы.

| Таблица | Назначение |
|---|---|
| `local_ikt_review_run` | Один запуск: период, статус, версия расчета, ошибка |
| `local_ikt_review_snap` | Сырые факты и агрегаты по курсу в рамках запуска |
| `local_ikt_review_metric` | Рассчитанные метрики по курсу, значение хранится как JSON |
| `local_ikt_review_log` | Логи шагов, длительность и число обработанных строк |

Основной поток данных:

```text
Moodle tables
  -> temporary tables
  -> local_ikt_review_snap
  -> metric classes
  -> local_ikt_review_metric
  -> live summary
```

Широкая таблица `local_ikt_review_snap` выбрана намеренно: при редких запусках
она позволяет пересчитывать формулы без повторного сканирования тяжелых
таблиц Moodle.

### Внутренние маркеры запусков

`local_ikt_review_run.calculationversion` остается техническим полем. Оно не
показывается в UI, но фиксирует версию расчета, которой был создан snapshot.
Это нужно для аудита после изменения формул или collect-логики.

Текущая production-версия расчета задается в `classes/manager.php` как
`collect-v4`.

Суффиксы `-selected` и `-synthetic` являются наследием временной debug/test
зоны. Они нужны только для того, чтобы старые тестовые прогоны, если они
остались в базе, не попадали в live-сводку и таблицу запусков. После
операторской очистки таких запусков из БД фильтры по этим суффиксам можно
убрать вместе с поддержкой выбранных course id, если она больше не нужна.

`local_ikt_review_log` не является debug-страницей. Таблица остается частью
production-диагностики: по ней можно понять, на каком шаге упал или замедлился
расчет. UI больше не показывает лог шагов напрямую.

## Production Pipeline

Оркестратор находится в `classes/manager.php`.

1. Создать `local_ikt_review_run` со статусом `queued`.
2. Поставить `task\calculation_task` в очередь Moodle ad hoc tasks.
3. В фоновом worker перевести запуск в `running` и получить Moodle lock.
4. Создать `tmp_ikt_review_courses`, добавить unique index `(courseid)` и
   выполнить `ANALYZE`.
5. Найти ID модуля `bigbluebuttonbn`.
6. Создать `tmp_ikt_review_course_students`.
7. Создать `tmp_ikt_review_log_filtered` и выполнить `ANALYZE`.
8. Создать агрегат `tmp_ikt_review_log_agg`.
9. Выполнить collect-шаги:
   `course_info`, `course_items`, `bbb_live`, `students`, `views`, `answers`,
   `grades`.
10. Рассчитать и сохранить метрики:
   `content`, `attendance`, `bbb`, `done`, `check`, `performance`.
11. Завершить запуск статусом `finished` или `failed`.

Каждый шаг работает в отдельной транзакции и получает
`statement_timeout = '30min'`. В `local_ikt_review_log` сохраняются длительность
и количество строк. Перед тяжелым действием также сохраняется `Step started`,
чтобы в БД было видно, на каком шаге находится незавершенный или упавший
запуск.

Очередь требует работающего Moodle cron worker. Если worker не запущен, запись
останется в статусе `queued`; через 24 часа UI пометит зависший запуск как
`failed`.

Переиспользуемая аналитика запуска находится в `classes/report/`. Ее использует
модальное окно подробностей на live-странице.

## Отбор курсов

Production-список курсов создается в `sql/collect/courses.sql`:

```sql
SELECT id AS courseid
  FROM {course}
 WHERE visible = 1
   AND idnumber LIKE '0%'
```

Это локальная конвенция текущей установки Moodle: дневные курсы имеют
`idnumber`, начинающийся с `0`.

### Известное ограничение

Фильтр не проверяет дерево `{course_categories}`. Перед использованием
статистики как официальной необходимо сверить множество `idnumber LIKE '0%'`
с веткой категорий `Дневная форма обучения` и определить правила для архивных,
служебных и ошибочно размещенных курсов.

## Дневные студенты

Рабочая роль дневного студента:

```text
full-time-student
```

`sql/collect/course_students.sql` создает пары `(courseid, userid)` только для
пользователей:

- записанных на курс через активный `{enrol}`;
- имеющих активную запись `{user_enrolments}`;
- имеющих роль `full-time-student` в контексте курса.

Эта temporary-таблица используется для:

- знаменателя `S`;
- фильтра просмотров;
- фильтра выполненных заданий и тестов;
- определения студенческих подключений к BBB.

### Известное ограничение

Текущая реализация считает состав студентов на момент запуска. Она пока не
учитывает `user_enrolments.timestart`, `user_enrolments.timeend`, а также
дополнительные признаки заблокированного или удаленного пользователя.
Повторный запуск старого периода после изменения состава курса может изменить
`S`, а значит и `At`, `Done`, `Check`.

## Котируемые элементы

Элементы считаются только если модуль и его секция открыты для студентов.

| Группа | Moodle modules |
|---|---|
| `Gr` - оцениваемые | `assign`, `quiz`, `vpl` |
| `T` - теория | `resource`, `page`, `lesson`, `book` |
| Диагностика, не входит в `T` | `url` |

Гиперссылки `url` сохраняются в snapshot для анализа, но не входят в формулу:
в исходном документе они отмечены как спорный тип.

## Формулы

### Content

```text
Content = 0.3 * T / Lectures + 0.7 * Gr / (PZ + LR)
```

В коде делители рассчитываются из часов:

```text
Lectures = lecture_hours / 2
PZ       = practice_hours / 2
LR       = lab_hours / 4
```

Пока интеграция с внешней системой не реализована,
`classes/api/api_provider.php` использует fallback:

```text
lecture_hours  = 36
practice_hours = 18
lab_hours      = 18
```

Следовательно, текущая fallback-формула:

```text
Content = 0.3 * T / 18 + 0.7 * Gr / 13.5
```

### Attendance

```text
At = View / (S * (Gr + T))
```

Где:

- `View` - количество просмотров котируемых элементов дневными студентами;
- `S` - число дневных студентов курса;
- `Gr + T` - число открытых котируемых элементов.

### Attendance Unique

```text
At_unique = UniqueView / (S * (Gr + T))
```

`UniqueView` считается как число уникальных пар `(userid, course_module_id)`
для дневных студентов и котируемых элементов. Это доля уникальных обращений
студентов к элементам курса.

### Live BBB

Курс считается имеющим живую ВКС, если в течение одного часа к одному BBB
подключились:

```text
не менее 1 преподавателя
и не менее 4 дневных студентов
```

То есть выполняется требование исходного документа: минимум 5 участников,
включая преподавателя.

### Done

```text
Done = Answer / (Gr * S)
```

`Answer` должен учитывать не более одного выполнения каждого оцениваемого
элемента каждым студентом.

#### Известное ограничение

Сейчас `answer_count` включает уникальные выполнения `assign` и завершенные
попытки `quiz`, но не включает выполнения `vpl`. При этом `vpl` входит в `Gr`.
Поэтому `Done` для курсов с VPL занижен. До исправления показатель нужно
интерпретировать с осторожностью.

### Check

Основной показатель:

```text
Check = graded_assign_submissions / submitted_assign_submissions
```

Учитывается не более одной отправки одного задания каждым дневным студентом.

Дополнительно сохраняется диагностический показатель:

```text
possible_check = graded_assign_submissions / (Assign * S)
```

### Performance

```text
Performance = average(course finalgrade)
```

Текущая реализация берет среднюю итоговую оценку из `{grade_grades}` для
`{grade_items}.itemtype = 'course'`.

#### Известное ограничение

`Performance` пока не фильтруется по периоду и по роли дневного студента.
Метрика сохраняется в snapshot, но не выводится в основной live-сводке.

## Итоговая сводка

Курс считается заполненным, если:

```text
Gr + T > 5
```

Это текущее бизнес-правило реализации. Его нет в исходном PDF, поэтому порог
нужно подтвердить отдельно.

Для всех курсов рассчитываются:

- количество курсов;
- количество и доля заполненных курсов;
- количество курсов с живыми ВКС.

Только по заполненным курсам рассчитываются средние:

- число котируемых элементов;
- `Content`;
- `At`;
- `At_unique`;
- `Done`;
- `Check`.

## Применение периода

`periodfrom` и `periodto` записываются в `local_ikt_review_run`.

Фильтруются по периоду:

| Источник | Поле |
|---|---|
| `{logstore_standard_log}` | `timecreated` |
| `{bigbluebuttonbn_logs}` | `timecreated` |
| `{assign_submission}` | `timemodified` |
| `{quiz_attempts}` | `timefinish` |

Читаются в текущем состоянии на момент запуска:

| Источник | Что влияет |
|---|---|
| `{course}` | видимость и metadata курса |
| `{course_modules}`, `{course_sections}` | набор открытых элементов |
| `{enrol}`, `{user_enrolments}`, `{role_assignments}` | состав студентов |
| `{grade_items}`, `{grade_grades}` | итоговые оценки |

Следствие: snapshot за прошлый период не является полностью исторически
воспроизводимым после изменения курса или состава студентов.

## Работа с Moodle logs

`log_filter.sql` читает `{logstore_standard_log}` за выбранный период:

```sql
l.timecreated BETWEEN :periodfrom AND :periodto
AND (l.contextlevel = 70 OR l.action = 'submitted')
```

На этом шаге еще не фильтруются дневные студенты и котируемые элементы. Это
намеренно: один тяжелый проход по логам материализует только потенциально
полезные события выбранных production-курсов. Количество строк в логе шага
`log_filter` - это размер временной таблицы `tmp_ikt_review_log_filtered`.

`log_aggregate.sql` затем соединяет эти события с:

- текущими дневными студентами из `tmp_ikt_review_course_students`;
- текущими открытыми котируемыми элементами из `{course_modules}` и
  `{course_sections}`.

Количество строк в логе шага `log_aggregate` - это число курсов, для которых
получился агрегат, а не число обработанных событий. При больших исторических
периодах почти все время ожидаемо уходит в `log_filter` и `log_aggregate`.

Temp-индексы на `tmp_ikt_review_log_filtered` не включены в production-проход:
на миллионах строк их создание может стоить дороже, чем выигрыш на агрегации.
Перед добавлением temp-индексов нужно отдельно проверять `EXPLAIN log_aggregate`
на сервере.

Для `log_aggregate` production локально устанавливает
`work_mem = '1GB'` через `SET LOCAL`. Это ограничено транзакцией шага и нужно,
чтобы тяжелые сортировки `COUNT DISTINCT` меньше уходили в `external merge` на
диске. Глобальные настройки PostgreSQL плагин не меняет.

## Collect SQL

| Файл | Назначение |
|---|---|
| `courses.sql` | Создает список production-курсов |
| `course_students.sql` | Создает пары дневных студентов курса |
| `log_filter.sql` | Один раз фильтрует тяжелую таблицу Moodle logs |
| `log_aggregate.sql` | Считает просмотры и уникальные просмотры дневных студентов |
| `course_info.sql` | Сохраняет metadata курса |
| `course_items.sql` | Считает открытые элементы курса |
| `bbb_live.sql` | Находит живые BBB по часовым окнам |
| `students.sql` | Сохраняет `S` |
| `views.sql` | Сохраняет обращения к элементам |
| `answers.sql` | Сохраняет выполнения assign и quiz |
| `grades.sql` | Сохраняет среднюю итоговую оценку |

Все collect-шаги обновляют одну строку курса через:

```sql
INSERT INTO {local_ikt_review_snap} (...)
SELECT ...
ON CONFLICT (runid, courseid)
DO UPDATE SET ...
```

## Индексы PostgreSQL

Плагин ожидает следующие дополнительные индексы:

| Таблица | Суффикс индекса | Назначение |
|---|---|---|
| `{logstore_standard_log}` | `lssl_ikt_time_course_v2_ix` | Фильтр логов за период |
| `{bigbluebuttonbn_logs}` | `bbbjoin_time_user_ix` | Подключения к BBB |
| `{assign_submission}` | `assub_sub_time_user_ix` | Выполненные задания |
| `{assign_grades}` | `asgr_valid_ass_user_ix` | Проверенные задания |
| `{quiz_attempts}` | `quiza_fin_time_user_ix` | Завершенные попытки |
| `{grade_grades}` | `gragr_final_item_ix` | Итоговые оценки |

Определения находятся в `db/upgradelib.php`.

### Важно для production

Текущий `db/install.php` и `db/upgrade.php` создают отсутствующие индексы
обычным `CREATE INDEX`. На больших таблицах это может блокировать запись.
Обновление плагина нужно выполнять в техническое окно.

Целевое улучшение перед регулярной эксплуатацией:

1. вынести создание индексов в операторский SQL или CLI;
2. создавать крупные индексы через `CREATE INDEX CONCURRENTLY`;
3. проверять наличие и определения индексов внешним SQL-чек-листом;
4. оставить в upgrade только безопасную валидацию.

## Известный технический долг

### Влияет на достоверность статистики

1. Уточнить состав `S`: сроки enrolment, deleted/suspended users и исторический
   состав курса.
2. Добавить выполненные VPL в `answer_count` или временно исключить VPL из
   знаменателя `Done`.
3. Сверить `idnumber LIKE '0%'` с деревом категорий дневной формы.
4. Подтвердить бизнес-порог заполненного курса `Gr + T > 5`.
5. Определить историческую семантику `Performance`.

### Влияет на эксплуатацию

1. Вынести создание больших индексов из upgrade.
2. Реализовать внешний источник нагрузки для `Content`.
3. После очистки старых тестовых запусков убрать legacy-фильтры
   `-selected`/`-synthetic`, если выбранные прогоны больше не нужны.
4. Добавить PHPUnit-тесты формул и fixture-тесты collect SQL.

## Проверка перед production-запуском

1. Выполнить upgrade в техническое окно.
2. Проверить ожидаемые индексы отдельным SQL на сервере.
3. Выполнить полный production-запуск.
4. Открыть подробности завершенного запуска на live-странице.
5. Сверить агрегаты и формулы вручную по выборке курсов.
6. Сравнить длительность шагов с предыдущими запусками.
